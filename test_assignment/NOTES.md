# NOTES

- **Task 1.** Normalization is grouped by `sku_norm` (UPPER + strip spaces/hyphens). Name picks the lexicographically first non-empty `name_norm` from duplicates to keep the choice deterministic; color uses `COLOR_MAP` with a case-insensitive fallback; `photo_url_best` is the first non-empty URL across duplicates. `issues_flag` is `OK` / `DUPLICATE` / `MISSING_ATTR` (DUPLICATE wins when both apply); the ISSUES sheet lists `MISSING_ATTR` separately so it stays visible.
- **Task 2.** Queries assume `payments.status='paid'` is the only successful state; under-payment query counts only paid payments toward `paid_amount`. Query #2 excludes orders without lines (they belong to query #1) and treats `|delta| > 1` as the threshold, matching the brief. `ROUND(..., 2)` is used for stable comparison output.
- **Task 3.** Money math uses `Decimal` with `ROUND_HALF_UP` to two decimals; HTML is generated from a single template with all user-supplied strings HTML-escaped (verified by a unit test). The OUTPUT sheet is laid out in three header/data blocks (partner / sale_order / sale_order_lines) plus a TOTAL row, matching the template structure.
- **Not done / out of scope.** No Power Query M-code is supplied for Task 1 — the same rules are expressed in `task1_normalize.py` instead. There is no DDL / sample data file for Task 2; SQL is validated against an in-memory SQLite fixture in `tests/test_task2_sql.py` (SQLite shares the relevant syntax with PostgreSQL).
